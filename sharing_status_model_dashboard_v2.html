Executive Summary

This dashboard documents the unified status tracking model for UAE PASS document sharing transactions. The model provides lifecycle-based visibility into every sharing request, replacing flat transaction counts with a detailed status timeline and clear funnel metrics per channel.

Purpose: Track where users progress or drop off in the sharing flow. Enable precise identification of bottlenecks and optimization opportunities across Notification, Redirect, and QR channels.

Channels Supported

Notification
Push notification flow
Redirect
App/Web-to-app redirect
QR
Web QR scan flow

Terminal S40 - S44 Final outcomes (success or failure)

Transition Rules Summary

Rule Description
Channel-specific ingress S01-S03 for Notification only; S04-S05 for Redirect only; S06-S07 for QR only
Convergence at S08 All channels must pass through S08 (REQUEST_VIEWED) before document check
Document branch S08 → S10 (docs ready) OR S08 → S11 (docs missing)
Missing doc outcomes S12 → S13 (success) | S14 (error, retry) | S15 (not found → S44)
Consent flow S20 → S21 → S22 → S30 (linear progression)
PIN outcomes S30 → S31 (verified) | S32 (failed → S43)
Expiry Any non-terminal → S42 when TTL reached
User abort Most states → S43 if user backs out
Single terminal Each request ends with exactly one terminal status (S40-S44)

Data Model & Schema

The status tracking system uses an event-sourced architecture with append-only status history.

Core Entities

-- sharing_requests (one row per request) CREATE TABLE sharing_requests ( request_id VARCHAR(64) PRIMARY KEY, -- Correlation ID sp_id VARCHAR(128), -- Service Provider channel ENUM('notification','redirect','qr'), platform ENUM('ios','android'), app_version VARCHAR(16), required_docs JSONB, -- Array of doc types created_ts TIMESTAMP ); -- sharing_status_history (1..n rows per request) CREATE TABLE sharing_status_history ( request_id VARCHAR(64), status_code VARCHAR(8), -- S00, S01, ... S44 status_ts TIMESTAMP, step_latency_ms INT, -- Since previous status total_age_ms INT, -- Since S00 meta JSONB, -- doc_context, channel data error_code VARCHAR(64), error_source VARCHAR(32), -- issuer, dv, network, user UNIQUE(request_id, status_code, status_ts) ); -- v_sharing_latest_status (view) CREATE VIEW v_sharing_latest_status AS SELECT DISTINCT ON (request_id) request_id, status_code AS latest_status, status_ts AS latest_ts FROM sharing_status_history ORDER BY request_id, status_ts DESC;

Event Schema (Analytics Logging)

{ "event_name": "sharing_status_changed", "request_id": "REQ-ABC123", "sp_id": "Botim", "channel": "notification", "platform": "ios", "app_version": "6.4.1", "from_status_code": "S21", "to_status_code": "S22", "doc_context": { "required_count": 2, "missing_count": 0, "missing_types": [] }, "error": null, "timing": { "event_ts": "2026-01-28T10:15:30Z", "step_latency_ms": 2500, "total_age_ms": 45000 } }

Identified Bottlenecks

Key drop-off points in the sharing flow based on sample data analysis.

1. Document Availability (S08 → S10/S11)
27% have missing docs at first view
135 requests affected

SPs requesting documents users don't have. Consider implementing document pre-check API.

2. User Abort at Consent (S20 → S43)
17.8% of all requests
89 users backed out

Users leaving at consent screen. Review UX clarity and SP purpose explanations.

3. Notification Channel Performance
59.7% success rate
23% below Redirect channel

Notification channel underperforms compared to Redirect (83%). Optimize notification timing and content.

4. Platform Gap (iOS vs Android)
68.3% iOS success
62.9% Android success

5.4 percentage point gap. Investigate Android-specific UX issues.

Failure Distribution

Recommendations

1

Implement Document Pre-Check API

Allow SPs to verify document availability before creating requests. Could eliminate ~27% of futile requests.

2

Improve Consent Screen UX

Clarify document list, add SP purpose explanations, reduce cognitive load. Target: reduce abort rate from 17.8% to <10%.

3

Notification Channel Optimization

Review notification content, timing, and engagement patterns. Benchmark against Redirect channel best practices.

4

Android Performance Sprint

Dedicated investigation of Android-specific drop-offs. Close the 5.4% platform gap.

Report Generation SQL Examples

Sample queries for generating key reports from the data.

Funnel by Channel

SELECT channel, COUNT(DISTINCT CASE WHEN status_code = 'S00' THEN request_id END) AS created, COUNT(DISTINCT CASE WHEN status_code = 'S08' THEN request_id END) AS viewed, COUNT(DISTINCT CASE WHEN status_code = 'S21' THEN request_id END) AS consent, COUNT(DISTINCT CASE WHEN status_code = 'S40' THEN request_id END) AS success FROM sharing_status_history GROUP BY channel;

SP Performance Scorecard

WITH request_outcomes AS ( SELECT request_id, sp_id, MAX(CASE WHEN status_code = 'S40' THEN 1 ELSE 0 END) AS is_success FROM sharing_status_history GROUP BY request_id, sp_id ) SELECT sp_id, COUNT(*) AS total_requests, SUM(is_success) AS successful, ROUND(AVG(is_success) * 100, 1) AS success_rate_pct FROM request_outcomes GROUP BY sp_id ORDER BY success_rate_pct DESC;

Step Latency Analysis

SELECT previous_status || ' → ' || status_code AS transition, COUNT(*) AS occurrences, ROUND(AVG(step_latency_ms)/1000, 2) AS avg_sec, ROUND(PERCENTILE_CONT(0.9) WITHIN GROUP (ORDER BY step_latency_ms)/1000, 2) AS p90_sec FROM sharing_status_history WHERE previous_status IS NOT NULL GROUP BY transition ORDER BY avg_sec DESC;
