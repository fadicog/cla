The Problem: Verified Data Stuck Behind Manual Entry

Current Reality
  • SPs ask users to manually re-enter data UAE PASS already holds as verified documents
  • A typical onboarding form: 8 to 15 fields the user has already proven to the government
  • Result: friction, drop-off, data-entry errors, and downstream re-verification costs for SPs
  • Users present documents for viewing then retype the same data into the SP's form
The paradox: UAE PASS verifies identity, delivers eSeal-backed documents to the SP — then the SP ignores the structured data and asks the user to type it again.

Layer 1 — Visible Claims (Convenience)

Auto-fill SP forms from document face data: name, DOB, EID number, nationality, expiry dates. Eliminates manual re-entry for users; reduces errors for SPs.

  • Immediate user convenience
  • Reduces onboarding time by minutes per session
  • Data is eSeal-verified — SPs skip re-verification
Layer 2 — Hidden Claims (Strategic)

Share issuer-signed fields NOT on the document face: WPS employer status, ISCO occupation code, EJARI number. A verified data channel SPs cannot get elsewhere.

  • Replaces 1-3 day manual verification steps
  • Positions UAE PASS as national verified data API
  • New value proposition for SP onboarding pipeline
UAE PASS is already the trust layer. Form Pre-Fill extends that trust from identity verification to verified data delivery.

Key Architecture Notes

  • Current model: document-level sharing via Verifiable Presentation
  • Pre-fill extracts specific fields (claims) from the VP payload
  • All shared data carries full eSeal provenance
  • Attribute-level API is exploratory for hidden claims (Phase 3)
Consent principle: User sees exactly which fields will be shared and from which document. No silent data extraction.

The Hidden Claims Opportunity

6 / 12
Strategic data layer
What are hidden claims? Issuer-signed fields included in the document's structured payload but NOT rendered on the visible document face. They carry full eSeal provenance — same trust as visible fields — but are invisible during normal document viewing.
WPS Employer Registration Status
Source: Labour Card (MoHRE)
Confirms employer is active in the Wages Protection System. Currently requires 1-3 day manual verification by banks for credit products.
ISCO Occupation Code
Source: Labour Card (MoHRE)
Standardised occupation code used globally for income-band estimation and credit scoring. Not visible on Labour Card face.
EJARI Unified Number
Source: Tenancy Contract (DLD)
Unique tenancy registration reference. Enables utility connection automation (DEWA) and insurance address verification.
Why this matters for TDRA
  • New product category: Verified data API, not just document sharing
  • No alternative channel: SPs cannot get this data from document inspection or user self-declaration
  • Replaces manual verification: 1-3 day bank processes reduced to seconds
  • B2G data infrastructure: Positions UAE PASS as national verified data backbone
Policy requirement: Each hidden claim category requires a distinct TDRA policy instrument. Users must explicitly see and approve non-visible data before sharing (new DDA consent UX pattern needed).
Recommendation: Present hidden claims as a separate strategic initiative with its own roadmap, not bundled into Form Pre-Fill Phases 1-2.

Decisions Needed from TDRA

11 / 12
Four decisions required to proceed
1

Data Sharing Authorisation Scope

Does the existing UAE PASS data sharing authorisation framework cover Form Pre-Fill (extracting specific claims from shared documents), or is a new policy instrument required?

Impact: Determines whether Phase 1 can proceed under current authorities or requires a new approval cycle.
2

Salary as a Claim Category

Is salary a permissible claim category for sharing with commercial SPs (banks, insurers)? Currently blocked pending TDRA decision.

Impact: Salary is the single highest-value claim for banking credit products. Decision unblocks or permanently excludes it.
3

Hidden Claims Policy Instrument

Should a separate policy workstream be initiated for hidden claims (non-visible issuer payload fields)? Each claim category would require its own authorisation.

Impact: Opens or closes the strategic "verified data API" opportunity — the highest-value differentiation for UAE PASS.
4

Phase 0 Discovery Approval

Approve initiation of Phase 0 discovery: Engineering VP payload audit, ICP schema request, and SP pilot outreach (Etisalat Retail + ADIB recommended).

Impact: Low-cost, high-information gate. Engineering audit is hours of work; results determine entire feature feasibility.
Note: Decisions 1-3 can proceed in parallel with Phase 0 discovery. Decision 4 is the immediate ask — it costs very little and provides the data foundation for all subsequent decisions.
