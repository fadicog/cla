Transaction Record Schema

Each sharing request generates multiple status records as it progresses through the flow. Here's the data structure:

// Sharing Transaction Record { "request_id": STRING, // Unique request identifier (e.g., "REQ000001") "sp_id": STRING, // Service Provider name "channel": ENUM, // "notification" | "qr" | "redirect" "platform": ENUM, // "ios" | "android" "app_version": STRING, // UAE PASS app version "required_docs": ARRAY, // List of requested document types "required_count": INTEGER, // Number of documents requested "status_code": ENUM, // Current status (S00-S44) "previous_status": ENUM, // Previous status code "status_ts": TIMESTAMP, // When this status was recorded "step_latency_ms": INTEGER,// Time since previous status (ms) "missing_count": INTEGER, // Number of missing documents "error_code": STRING, // Error code if applicable "error_source": STRING, // Error origin (user, issuer, system) "status_history": ARRAY // Ordered list of all statuses }

Example Transaction Flow

// Successful sharing via notification channel "REQ000002" Status History: S00 → S01 → S02 → S03 → S08 → S10 → S20 → S21 → S22 → S30 → S31 → S40 // Failed - user declined consent "REQ000006" Status History: S00 → S01 → S02 → S03 → S08 → S10 → S20 → S21 → S30 → S32 → S43 // Failed - documents not available "REQ000004" Status History: S00 → S01 → S02 → S03 → S08 → S11 → S12 → S15 → S44

Key Data Fields Explained

Field Purpose Analytics Use
channel How the user was reached Compare conversion rates across notification, QR, redirect
platform User's device OS Identify iOS vs Android performance gaps
step_latency_ms Time between status changes Identify slow steps, timeout risks
missing_count Documents user doesn't have Predict failure likelihood, document availability analysis
error_code Specific failure reason Root cause analysis for failures
status_history Complete journey record Path analysis, drop-off identification

Identified Bottlenecks

Analysis of where users drop off in the sharing flow and potential causes.

1. Consent Screen Drop-off (S20 → S21)
Impact: 30 requests (6.0%) exit at consent screen
Possible Causes: Complex consent UI, unclear document list, privacy concerns
Recommendation: Simplify consent screen design, add trust indicators
2. Document Availability (S08 → S10/S11)
Impact: 135 requests (27.0%) have missing documents
Possible Causes: SPs requesting documents users don't have
Recommendation: Implement document pre-check API for SPs
3. Consent to PIN (S30 → S31)
Impact: 16 requests (3.2%) fail at PIN entry
Possible Causes: Forgotten PIN, biometric fallback issues
Recommendation: Improve PIN recovery flow, enhance biometric options
4. Consent Declined (S43)
Impact: 89 requests (17.8%) explicitly declined
Possible Causes: Users not understanding why SP needs documents
Recommendation: Require SPs to provide clear purpose explanation

Failure Distribution

Priority Recommendations

1

Implement Document Pre-Check API

Allow SPs to verify document availability before creating sharing requests. This could eliminate ~27% of futile requests where users don't have required documents.

2

Redesign Consent Screen UX

Simplify the consent interface with clearer document listings and SP purpose explanations. Target: reduce consent decline rate from 17.8% to under 10%.

3

Android Performance Sprint

iOS shows 68.3% vs Android 62.9% success rate. Investigate Android-specific UX issues and optimize accordingly.

4

Notification Channel Optimization

Notification channel has lowest success rate (59.7%) vs Redirect (83.0%). Improve notification content and timing to increase engagement.
